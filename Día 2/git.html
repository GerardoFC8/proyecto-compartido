1. Conceptos Fundamentales

Git es un sistema de control de versiones distribuido, lo que significa que cada desarrollador tiene una copia completa del historial del proyecto. GitHub es una plataforma en la nube que utiliza Git para proporcionar un lugar centralizado para almacenar repositorios y colaborar.

Repositorio (Repo)

Una carpeta que Git rastrea, conteniendo todos los archivos del proyecto y su historial.

Commit

Una "instantánea" del repositorio en un momento dado, con un mensaje descriptivo.

Rama (Branch)

Un puntero a un commit que permite trabajar en funcionalidades de forma aislada.

HEAD

Un puntero que indica la rama en la que te encuentras trabajando actualmente.

2. Configuración Inicial de Git

Antes de empezar, es crucial configurar tu identidad en Git. Estos datos se adjuntarán a cada uno de tus commits.

git config --global user.name "Tu Nombre"
git config --global user.email "tu.email@ejemplo.com"

3. Flujo de Trabajo Básico (Local)

3.1. Creación de un Repositorio

Para iniciar un nuevo repositorio en tu máquina local:

# Navega a la carpeta de tu proyecto
cd /ruta/a/mi/proyecto

# Inicializa Git en la carpeta
git init

3.2. Realización de Commits

Un commit es el paso fundamental para guardar tus cambios. Sigue este flujo:

Working Directory

Modificas archivos

→

Staging Area

`git add`

→

Local Repo

`git commit`

Prepara los cambios (staging) y luego haz el commit.

# Añade todos los archivos modificados
git add .
# Guarda los cambios con un mensaje
git commit -m "Descripción concisa de los cambios"

4. Colaboración con GitHub

4.1. Conexión y Subida de Cambios (Push)

Después de crear un repositorio en GitHub, conéctalo a tu repositorio local y sube tus commits.

# Enlaza tu repo local con el remoto de GitHub
git remote add origin https://github.com/tu-usuario/nombre-del-repo.git

# Renombra la rama principal a 'main' si es necesario
git branch -M main
# Sube la rama 'main' al remoto 'origin'
git push -u origin main

4.2. Descarga de Cambios (Pull)

Para obtener los cambios más recientes del repositorio remoto y fusionarlos con tu trabajo local.

# Descarga y fusiona los cambios de la rama remota 'main'
git pull origin main

5. Flujo de Trabajo en Equipo

El uso de ramas es crucial para el trabajo en equipo. La rama `main` debe ser siempre estable y funcional. Este es el flujo de trabajo recomendado para desarrollar una nueva funcionalidad.

1. Crear Rama

`git checkout -b feature/algo`

→
2. Desarrollar

`git add .` `git commit`

→
3. Subir Rama

`git push origin feature/algo`

4. Abrir Pull Request

En GitHub, para revisión

→
5. Fusionar (Merge)

Incorporar cambios a `main`

5.4. Resolución de Conflictos

Los conflictos ocurren cuando dos desarrolladores modifican la misma línea. Git te pedirá que resuelvas el conflicto manualmente. Verás algo como esto en tu editor:

<<<<<<< HEAD

const color = "blue"; // Tu cambio

=======

const color = "green"; // Cambio del otro desarrollador

>>>>>>> some-other-branch-name

Debes editar el archivo para quedarte con el código correcto, eliminar los marcadores de Git (`<<<`, `===`, `>>>`), y luego hacer un commit con la resolución.

# Después de editar el archivo y resolver
git add archivo_con_conflicto.js
git commit -m "Resolución de conflicto de color"

6. Comandos Avanzados

Clonar un repositorio

git clone https://...

Ver el estado

git status

Ver historial de commits

git log --oneline

Revertir un commit (seguro)

git revert <hash-commit>

7. Consejos para un Flujo de Trabajo Sin Conflictos

  • Trabaja en ramas separadas: Nunca trabajes directamente en `main`.
  • Haz `pull` a menudo: Descarga los cambios de `main` antes de empezar a trabajar.
  • Commits atómicos: Cada commit debe contener un solo cambio lógico.
  • Mensajes de commit claros: Describe qué hiciste y por qué.
  • Revisión de código: Fomenta la revisión por pares a través de Pull Requests.
  • Utiliza `.gitignore`: Ignora archivos generados automáticamente (logs, `node_modules`, etc.).
